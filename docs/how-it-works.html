How the arena runs a race

Two coding agents work the same practice-puzzle chain, each in its own container, and every step streams to the frontend. This page walks the five flows that make one run happen, straight from the code.

00

The stack

One Node process runs the whole arena. Fastify serves the HTTP and SSE routes; RunManager owns the run lifecycle and the two drivers. Docker isolates the entrants, one hardened container each, and each container carries the burner's WALLET_ADDRESS, WALLET_PRIVATE_KEY, and ETH_RPC_URL as env. SQLite is the only store: events, run and entrant rows, burner keys, and scores. A private hardhat or anvil chain at chainId 31337 is the game board. The Vite/React frontend holds no state of its own; it watches the event stream.

The chain hosts only practice puzzles. Each puzzle is a small Solidity contract with an intended solution, like an advent-of-code box. Nothing here is a live system.

What lives in SQLite

One file, six tables. Every burner key and every event row sits here in plaintext; it is a local demo store, wiped by deleting the file. src/db/schema.ts

01

Wallet generation and funding

Starting a run walks a state machine. Each step is a legal transition; an illegal one throws. The wallet and funding work happens between preparing and ready, so an agent never boots without a funded burner to sign with.

How the burner is born

In the preparing phase the wallet gate runs first, before any container exists. It makes one burner per entrant so the address is known when the container gets its environment.

  • The gate loops over both entrants and calls createWallet, which does generatePrivateKey() from viem, derives the address, and writes both to the wallets table. src/chain/wallet.ts
  • It writes the address back onto the entrant row (entrants.address) and emits a wallet.assigned event on that entrant's lane. src/chain/funding-gate.ts · createWalletGate
  • The private key stays in SQLite and later rides into the container as an env var. It is never emitted as an event.

How the burner gets funded

After both containers prepare, the run moves to awaiting_funding and the funding gate runs. It starts the balance watcher, then funds the burners, and holds the run until both are funded.

  • The watcher awaitFunding starts first and polls every 500 ms. It reads each balance at head − confirmations, not at head. src/chain/funding-watcher.ts
  • Funding sends 0.1 ETH to each burner from Hardhat dev account 0 (the well-known local key). It only runs when the profile is local and the chain id is 31337. src/chain/funding-gate.ts · fundLocalEntrants
  • It then mines one block with evm_mine. The local profile sets confirmations: 1, so mining one block moves the funding transaction to head − 1 where the watcher can see it. config/chains.json
  • Every time a balance changes, the watcher emits a funding.balance event with the wei amount and a funded flag. The gate resolves once both burners hold at least 0.05 ETH, then the run moves to ready.
Two facts the summary left out. Both the wallet gate and the funding gate return early unless the preset is docker-duel. The fake-duel preset skips wallets and funding entirely and streams canned events. And the funder key and profile are overridable through ARENA_FUNDER_KEY and ARENA_CHAIN_PROFILE; the Hardhat account is the default, not a hard-coded constant. src/chain/funding-gate.ts
02

Credentials into each container

Each entrant runs in its own hardened container. Before the container starts, the driver builds a temporary credential directory on the host and mounts it in. Login files go in over a mount; the wallet and the RPC URL go in as environment variables.

codex

  • Makes a fresh temp dir, copies the operator's auth.json in, and writes a config.toml. It pins a model only when the preset names one; a ChatGPT-account login rejects API-only model ids, so the default login stays on its own default. src/adapters/codex.ts · createContainer
  • Mounts that dir at /creds/codex and sets CODEX_HOME to the same path so the CLI finds the login.

opencode

  • Makes a temp dir mounted at /creds/opencode. The OpenRouter key comes from OPENROUTER_API_KEY or the operator's opencode auth.json, and rides in as the OPENROUTER_API_KEY env var. src/adapters/opencode.ts
  • OPENCODE_SERVER_PASSWORD and OPENCODE_PORT get stripped from the container environment before launch. scrubOpenCodeEnvironment

Wallet injection (both harnesses)

  • The driver reads the burner from SQLite with getWallet and, when it exists, adds WALLET_ADDRESS and WALLET_PRIVATE_KEY to the container env alongside ETH_RPC_URL. The agent signs with cast send --private-key "$WALLET_PRIVATE_KEY".

Why the container is boxed in

  • CapDrop: ['ALL'] and SecurityOpt: ['no-new-privileges']: no Linux capabilities, no privilege escalation. Never privileged, never mounts the Docker socket. src/runtime/container.ts
  • Bounded resources: PidsLimit 512, Memory 2 GiB, NanoCpus 2. One private bridge network per entrant, and its own credential mount.
  • ExtraHosts: ['host.docker.internal:host-gateway'] lets the container reach the chain running on the host. ETH_RPC_URL points at http://host.docker.internal:8545.
Why not localhost:8545. Inside a container, localhost is the container itself, not the host. The chain runs on the host, so the container reaches it through host.docker.internal, which the host-gateway entry resolves. The opening prompt tells the agent this outright. src/run-manager.ts · buildOpeningPrompt
03

Server-side events and the SSE feed

Every fact in a run is one row in an append-only journal: a log the code only adds to, never edits. The frontend reads that journal live over one SSE stream per run, and a reconnect picks up where it left off with no gap and no duplicate.

SSE is Server-Sent Events: one ordinary HTTP response the server never closes. It keeps appending lines as events happen, and the browser's built-in EventSource reads them live. On a dropped connection EventSource reconnects on its own and resends the last id it saw in a Last-Event-ID header, so the server replays only what the browser missed. That free resume is why SSE fits this one-way feed better than a websocket, with no second protocol to run.

The journal

  • append assigns two numbers in one transaction: a global autoincrement id shared across all runs, and a seq that counts up per (runId, source). src/journal.ts
  • The id is one counter over every run, so inside a single run it can jump when another run writes in between. The seq counts 1, 2, 3… for one (run, source) with no holes. Section 4 leans on that difference.
  • source is the entrant id for a lane event, or the literal 'run' for a run-level event like a state change.
  • After the write, append pushes the event to any live subscriber for that run. after(runId, afterId) reads back every row past a given id, in id order.

The SSE endpoint

  • GET /runs/:id/events resumes from a cursor: it takes the higher of the ?after query and the Last-Event-ID header. src/server.ts · openEventStream
  • Replay then live, with no lost event in the seam: it subscribes first and buffers arriving events in a pending list while it replays history, then sorts the buffer by id and flushes it. send drops anything at or below the last id it wrote, so the overlap never double-sends.
  • It writes each event as id: <n> + data: <json>, and sends a : heartbeat comment every 15 s to hold the connection open.

  2. The container runner wraps that line as {ev:"line"} and sends it up the attachment stream. src/runtime/container.ts
  3. The driver hands the line to the opencode parser, which turns it into a tool.call event. src/adapters/opencode-parser.ts · runTurn in harness-driver.ts

04

How the frontend syncs

The frontend folds the event stream into one state object. It dedupes reconnect overlap, flags real data loss, and renders every event type as a line, so the feed never blanks.

Dedupe and gap detection

  • Dedupe runs on the global id. After a reconnect the browser replays a few events it already saw; seenIds recognizes those ids and drops them, so nothing counts twice. Returning the same state object lets React skip a re-render. packages/mock-frontend/src/feed-projection.ts
  • Loss detection runs on seq, not id, because id can skip for an innocent reason (another run wrote in between) while a source's seq never does. If opencode-1 jumps from seq 4 to seq 7, two events are gone: the UI records a FeedGap and shows a warning instead of a silently wrong race.
  • This is why both numbers exist. Without the id check a wifi blip could double-count a minted flag; without the seq check it could hide one, and the scoreboard would lie.

Lanes and the wallet header

  • eventsForSource filters the feed by source === entrantId, so each lane shows only its own entrant's events. Run-level events use the 'run' source.
  • describeEvent renders a one-line summary for every known type, and any type the UI doesn't know falls through to rawFallback, which prints the raw payload. The feed never shows a blank row.
  • deriveLaneWallet folds a lane's events into its wallet header. The address starts from the run snapshot, then wallet.assigned and funding.balance override it; balance and the funded flag come from the latest funding.balance. awaitingFunds is true while the run is awaiting_funding and this lane is not funded yet.
05

How steer works

An operator can inject a free-text turn into a live agent mid-race. If the agent is mid-turn, the steer waits in a queue and lands when the turn ends. If it's idle, it resumes the same harness session inside the same container.

  • POST /runs/:id/entrants/:entrantId/steer takes free text, replies 202, and calls RunManager.steer, which looks up the entrant and calls the driver. src/server.ts · src/run-manager.ts
  • A degraded entrant rejects the steer: the driver logs an entrant.error and returns. An entrant is stopping rejects too. src/adapters/harness-driver.ts · steer
  • If a turn is in flight, the text goes onto queuedSteers and the call returns. When the turn ends, finishTurn shifts one queued steer and starts it as a resume.
  • An idle entrant resumes right away. The resume runs codex exec … resume <sessionId> or opencode run -s <sessionId> in the same container, so the agent keeps its context. A resume before the harness has reported a session id marks the entrant degraded.
  • Each resume appends an entrant.steered event to that entrant's lane, so the steer shows up in the feed.
